## 5. Provision the Vercel project for `cloud.agorio.dev`

The cloud app lives at `C:\dev\agorio\cloud`. Create a new Vercel project pointing at that subdirectory:

1. Vercel dashboard → Add New → Project → Import from Git (existing `Nolpak14/agorio` repo).
2. **Root Directory:** `cloud`.
3. **Framework Preset:** Next.js (auto-detected).
4. **Build Command:** `npm run build` (default).
5. **Output Directory:** `.next` (default).
6. **Install Command:** `npm install` (default).

### Environment variables to set (Production + Preview)

| Name | Value | Notes |
|---|---|---|
| `DATABASE_URL` | from existing `site` project | Same Neon connection string used by `site` |
| `NEON_AUTH_BASE_URL` | from existing `site` project | Same Neon Auth instance — sessions are shared by email |
| `NEON_AUTH_COOKIE_SECRET` | from existing `site` project | Same secret so cookies from `agorio.dev` work on `cloud.agorio.dev` once domains share a parent |

You can pull these from the existing `agorio-site` project: Settings → Environment Variables → copy values.

### Domain wiring

1. Vercel project → Settings → Domains → Add `cloud.agorio.dev`.
2. Vercel shows the required DNS record (typically a CNAME to `cname.vercel-dns.com`).
3. Add the CNAME in your DNS provider for the `cloud` subdomain.
4. Wait for cert provisioning (usually <5 min).

### Cross-subdomain session sharing (CONFIRMED REQUIRED)

For Neon Auth cookies to authorize the same user on both `agorio.dev` and `cloud.agorio.dev`, **both** `site/lib/auth-server.ts` and `cloud/lib/auth-server.ts` must pass `cookies.domain: '.agorio.dev'` (gated on `VERCEL_ENV === 'production'` so localhost / preview keeps working). Already wired in v0.6.0.

### Neon Console — add `cloud.agorio.dev` to Trusted domains (REQUIRED)

The Neon Auth backend's `better-auth` instance enforces a server-side trusted-origins allow-list. Each subdomain that uses the same Neon Auth project must be added to that list via the Neon Console — there's no way to configure this from the app code (`createNeonAuth({ baseUrl, cookies })` has no `trustedOrigins` field).

**Without this step, sign-in / sign-up requests from the new subdomain will return `{"code":"INVALID_ORIGIN","message":"Invalid origin"}` 403.**

1. Open https://console.neon.tech
2. Open the project hosting the `neondb/auth` endpoint
3. Open the **Auth** section → **Project Info** → **Domains**
4. Add the subdomain URL (e.g. `https://cloud.agorio.dev`) under "Trusted domains"
5. Save

If anything fails, check:
- **Trace doesn't appear** → look at agent console for `[agorioCloud]` warnings.
- **401 from /api/ingest** → key revoked / cache stale (wait 60 s) / malformed.
- **403 INVALID_ORIGIN from /api/auth/...** → `cloud.agorio.dev` missing from Neon Console "Trusted domains". See §5.
- **"An unexpected response was received from the server" on Create Key** → middleware matcher is intercepting the action POST. Remove the page route from `cloud/middleware.ts` matcher (it should only cover non-action routes like `/traces/:path*`).
- **Row stays "in_progress"** → agent didn't run `onComplete`; check Node version (≥18 for `crypto.randomUUID`).
- **404 on /api/auth/sign-in/email** → env var corruption. See "Operational gotchas" below.

## Rollback playbook (if Cloud breaks for paying customers)

1. **Cloud dashboard is down but ingestion works** → users keep agenting; investigate Vercel/Neon side. No data loss.
2. **Ingestion endpoint is rejecting valid keys** → check the in-memory cache TTL (60 s); restart cloud deployment to drop cache; verify `apiKeys` table reads work from Vercel logs.
3. **Schema migration left bad state** → Neon supports branching; create a branch from a pre-migration timestamp and re-point `DATABASE_URL`. Do NOT `drop table` in production.
4. **SDK 0.6.0 broke downstream consumers** → `npm deprecate @agorio/sdk@0.6.0 "regression — use 0.5.0"`; users pin back to 0.5.0. The 0.6 SDK additions are purely additive, so a downgrade is safe unless they've adopted `onComplete`.

## Operational gotchas (learned the hard way)

These bit us during the v0.6 launch. Documenting so the next person doesn't repeat them.

### Don't round-trip env vars through `vercel env pull`

The `vercel env pull` command writes values in JSON-style escaped form:

```
KEY="value containing\nliteral escapes"
```

The `\n` in the output is a literal two-character sequence. If you bulk-copy these values to another project via `vercel env add` without de-escaping, the destination project's env value ends up with literal `\n` characters appended. Server-side code then builds URLs like `…/auth\n/sign-in/email`, getting 404 from the upstream.

**Right way to copy env vars between Vercel projects:**

Either copy values manually via the Vercel UI, or source them from the canonical `.env.local` (which uses single quotes and no escapes):

```bash
NA_BASE=$(grep "^NEON_AUTH_BASE_URL=" /path/to/site/.env.local | sed -E "s/^NEON_AUTH_BASE_URL=['\"]?(.*)['\"]?$/\1/" | sed "s/'$//")
printf '%s' "$NA_BASE" | vercel env add NEON_AUTH_BASE_URL production --force
```

After copying, verify by pulling back and inspecting the head + tail of each value:

```bash
vercel env pull .env.check --environment=production --yes
grep "^NEON_AUTH_BASE_URL" .env.check | sed -E 's/^([A-Z_]+)="(.{6}).*(.{6})"/\1: head=\2 tail=\3/'
# Should print: head=https: tail=b/auth
# NOT: head=https: tail=auth\n
```

### Don't put server-action routes in middleware matchers

Next.js 15 sends server-action bodies as `POST /<page-route>`. If Neon Auth middleware is matched against that route, its response augmentation makes the action client see "An unexpected response was received from the server."

Page-level auth via `getCurrentCustomer()` and action-level auth via `auth.getSession()` are sufficient. Only put GET-only routes in the middleware matcher.

### A new subdomain on a shared Neon Auth project needs Neon Console config

Every subdomain that talks to the same Neon Auth backend must be added to the project's "Trusted domains" list via the Neon Console. Not configurable from app code. See §5.

### Manual customer comping for owner / beta accounts

For testing without going through Stripe checkout (e.g., the project owner trying their own product), you can comp a customer row directly:

```js
import { neon } from '@neondatabase/serverless';
import { randomBytes } from 'node:crypto';

const sql = neon(process.env.DATABASE_URL);
const email = 'you@example.com';
const licenseKey = 'agorio_pro_' + randomBytes(16).toString('hex');
const stripeId = 'manual_comp_' + Date.now() + '_' + randomBytes(4).toString('hex');

await sql`
  INSERT INTO customers (stripe_customer_id, email, license_key, status, plan)
  VALUES (${stripeId}, ${email}, ${licenseKey}, 'active', 'enterprise')
`;
```

The `stripeId` prefix `manual_comp_` makes these rows easy to identify (`SELECT … WHERE stripe_customer_id LIKE 'manual_comp_%'`).

**Caveat:** the Stripe webhook does not check for existing rows by email, so if you later sign up via real Stripe checkout with the same email, you'll end up with two rows. Either delete the comp row first, or fix the webhook to upsert on email (tracked for v0.6.1).
