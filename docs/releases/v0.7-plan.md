# Agorio v0.7 — B2B Procurement Vertical Plan

> **Status:** in progress, May 2026. Tracks [GitHub issue #37](https://github.com/Nolpak14/agorio/issues/37).
>
> **Target:** Q3/Q4 2026 (~6 weeks)
> **Theme:** Ship the headline B2B procurement demo (the marketing wedge) and the primitives that make it non-trivial to clone (the technical moat).

## Context

v0.6 shipped the hosted Cloud control plane — `agorioCloud({ apiKey })` ships traces to `cloud.agorio.dev`, customers can authenticate, mint API keys, and see per-run drilldowns. The Pro tier is unlocked by Cloud access.

v0.7 builds on that foundation to ship the procurement vertical. Two parallel goals:

1. **Marketing wedge.** A reference agent that B2B sales conversations can lead with — multi-merchant price comparison, human-in-the-loop approval, PO# tracking, expense categorization, full audit trail surfaced in Cloud. Procurement is the highest-WTP agentic-commerce use case and Agorio's plugin system was already most of the way there.
2. **Technical moat.** Primitives that make the demo non-trivial to clone — sub-agent composition with first-class Cloud span hierarchy, durable session resume across approval pauses, and the procurement-specific governance plugin. Competitors can copy a single-agent demo in a weekend; they can't copy a multi-agent run with resumable state and full audit trail.

## Goals

1. Ship retry + rate-limit HTTP primitives so multi-merchant comparison doesn't trip Storefront throttles
2. Sub-agent invocation primitive + `AgentChain` orchestrator
3. `parent_span_id` convention so the Cloud trace explorer renders multi-agent runs as a tree
4. `SessionStorage` interface + `MemorySessionStorage` + `FileSessionStorage` in-tree
5. Separate `@agorio/session-redis` package proving the interface is real
6. New `@agorio/plugin-procurement` package (sixth governance plugin)
7. `examples/procurement/` reference agent with CI smoke test
8. `/procurement` landing page + README section

## Out of scope (deferred)

- Cloud-backed `SessionStorage` (writes through the existing ingestion endpoint) — v0.7.1
- Hosted approval-workflow receiver on `cloud.agorio.dev/api/approve/<id>` — v0.7.1 (was on v0.6.1)
- Real-merchant connector for the full demo mode (Shopify dev store + WooCommerce docker) ships as docs + setup guides; the core example runs against MockMerchants for CI reproducibility
- Demo video and case study — out-of-band marketing work, tracked separately

## Implementation

### Step 1 — HTTP primitives: retry + rate limit (~3 days, **shipped**)

**New files:**
- `src/http/retry.ts` — `withRetry(fetchFn, opts)` exponential backoff with `Retry-After` header support
- `src/http/rate-limit.ts` — `TokenBucket` + `withRateLimit(fetchFn, bucketOrFactory)` (per-origin memoization)
- `src/http/index.ts` — `createHttpClient({ retry?, rateLimit? })` factory

**Integration:** drop-in via the existing `fetch?: typeof globalThis.fetch` option on `ShopifyAdapter`, `WooCommerceAdapter`, `UcpClient`, `AcpClient`, `McpClient`. No constructor signature changes.

**Tests:** `tests/http-retry.test.ts` (9 tests), `tests/http-rate-limit.test.ts` (6 tests).

### Step 2 — Sub-agent + AgentChain (~5 days, **shipped**)

**New types** (`src/types/index.ts`): `SubAgent`, `SubAgentBuildContext`, `AgentChainStep`, `ChainContext`, `ChainResultEntry`. New `subAgents?`, `subAgentMaxDepth?` options on `AgentOptions`.

**New files:**
- `src/agent/sub-agent.ts` — `runSubAgent({ subAgent, input, parentTracer?, parentOnLog?, ... })`. Generates a UUID per invocation; wraps the parent's tracer/onLog to inject `parent_span_id`, `sub_agent_name`, and `sub_agent_invocation_id` on every child span and log event. Recursion guard via `parentDepth` + `maxDepth` (default 3).
- `src/agent/agent-chain.ts` — `AgentChain` class with `.add(subAgent, inputFrom?)` and `.run(initialInput, options)`. Halts on first child failure; aggregates `AgentResult.usage` across steps.

**ShoppingAgent changes:**
- Auto-registers an `invoke_sub_agent` tool when `options.subAgents` is non-empty
- New private `toolInvokeSubAgent` handler delegates to `runSubAgent`
- Depth tracked via `_subAgentDepth` (internal) so chained invocations enforce the global cap

**Cloud trace explorer** (`cloud/app/traces/[runId]/page.tsx`):
- New `<SubAgentsStrip>` chip strip listing each sub-agent invocation, depth, and duration
- `<SpansTable>` indents child rows by `attributes.depth` (computed from `parent_span_id` / `sub_agent_name`)
- Wrapper `agent.sub_agent` rows rendered in the accent color so the run tree is scannable

**Tests:** `tests/sub-agent.test.ts` (6 tests), `tests/agent-chain.test.ts` (5 tests).

### Step 3 — Persistent sessions (~5 days, **shipped**)

**New types** (`src/types/index.ts`): `SessionState`, `SessionStorage`. New `sessionStorage?`, `sessionId?`, `sessionCustomerId?` options on `AgentOptions`. New `hydrate?(state)` hook on `EnterprisePlugin` (paired with the existing `getState?()`).

**New files in `src/session/`:**
- `memory-storage.ts` — `MemorySessionStorage` (Map-backed, dev/test only; uses `structuredClone` for isolation)
- `file-storage.ts` — `FileSessionStorage` (JSON files under a configurable dir, atomic temp+rename writes, path-traversal-safe sessionId regex)
- `index.ts` — barrel

**New package `packages/session-redis/`:**
- `@agorio/session-redis` exporting `RedisSessionStorage` with `RedisClientLike` adapter surface (ioredis / node-redis v4 compatible)
- Per-session TTL support, customer secondary index via Redis SET, mget batching for `list()`
- Own test suite (4 tests using an in-memory client stub)

**ShoppingAgent changes:**
- `tryHydrate()` runs at the top of `run()`; if a snapshot exists for the configured `sessionId` it restores `iteration`, conversation `messages`, `merchants` map, active merchant, and per-plugin state (via each plugin's `hydrate(getStateOutput)`)
- `persistSession()` runs after each iteration's tool execution loop; non-blocking (errors emit a warn log but never break a running agent)
- Adapter instances are NOT serialized — at hydrate time we re-resolve adapters by domain from `options.adapters[]`

**approval-workflow plugin:** implements `hydrate()` so pending approval requests survive a process restart — the headline procurement feature.

**Tests:** `tests/session-storage.test.ts` (13 tests covering both storages + a ShoppingAgent resume integration test), `packages/session-redis/tests/redis-storage.test.ts` (4 tests).

### Step 4 — `@agorio/plugin-procurement` (~4 days, **shipped**)

**New package:** `plugins/procurement/` mirroring `plugins/spending-controls/` layout exactly (own `package.json` with `peerDependencies: { "@agorio/sdk": "^0.7.0" }`, MIT LICENSE, dedicated test suite).

**Public API:** `createProcurementPlugin(config)` with:
- `vendors: VendorConfig[]` — id, name, domain, paymentTerms, taxCategory, preferred, metadata
- `expenseCategories?: string[]` — validated values for `categorize_expense`
- `poNumberPrefix?: string` (default `'PO'`)
- `poNumberStrategy?: 'sequential' | 'uuid' | () => string` (default sequential)
- `requirePoOnCheckout?: boolean` — when true, blocks `submit_payment` until a PO# is assigned
- `onProcurementCompleted?: (e: ProcurementCompletedEvent) => void`

**Tools exposed:** `assign_po_number`, `lookup_vendor`, `categorize_expense` (the LLM picks via a single `procurement` tool with an `action` enum, matching the `approval_workflow` plugin's convention).

**Hooks:** `onBeforeToolCall` blocks `submit_payment` when PO required and missing. `onAfterToolCall` fires `onProcurementCompleted` with PO# + vendor + category + amount on successful checkout. `getState` / `hydrate` round-trip cart tags so session resume preserves the PO# / vendor / category.

**Tests:** `plugins/procurement/tests/procurement.test.ts` (12 tests — PO strategies, vendor lookup, category validation, requirePoOnCheckout gate, audit-event emission, hydrate round-trip).

## Risks & mitigations

- **Sub-agent loops.** A misconfigured `AgentChain` could recursively invoke itself. Mitigation: hard cap of 3 nested depth, enforced in `runSubAgent()` (`parentDepth + 1 > maxDepth` throws). Test coverage in `tests/sub-agent.test.ts`.
- **Redis as an optional package.** Users get a confusing error if they import `@agorio/session-redis` without installing `ioredis`. Mitigation: separate published package with `ioredis` as a clear peerDep; README documents the dependency. Core SDK stays Redis-free.
- **Shopify dev store rate limits during demo.** Storefront API throttles at 2 req/s. Mitigation: the rate-limit primitive from Step 1 is wired into the Shopify adapter via `createHttpClient({ rateLimit })` with a conservative default for `*.myshopify.com`.
- **Procurement plugin API surface shifts on design-partner feedback.** Mitigation: ship as `0.1.0`, call it "preview" in README, reserve breaking-change freedom for 0.1.x.
- **Cloud span tree rendering complexity for large runs.** Tree-rendering 100+ spans can be slow in the browser. Mitigation for v0.7: indentation-only (flat list with depth-based padding) — full collapsible tree deferred to v0.8 when fleet view lands.
- **Session state growth.** Long agent runs with verbose conversation history bloat snapshots. Mitigation: documented in the SessionStorage interface JSDoc; a future `compactor` hook can trim message history at save time. Not in scope for v0.7.

## What v0.7 unblocks

- **v0.7.1 — Cloud-backed `SessionStorage`.** Once the interface is real and consumed by the procurement plugin, adding a `CloudSessionStorage` that writes through the existing ingestion endpoint is contained.
- **v0.7.1 — Hosted approval-workflow receiver.** The procurement demo runs a local receiver; promoting it to `cloud.agorio.dev/api/approve/<requestId>` is contained now that sessions can survive the wait.
- **v0.8 — EU AI Act audit export.** Procurement plugin emits structured `procurement_completed` events; the export module just serializes the existing `trace_spans` + `trace_logs` filtered by customer + date range.
- **v1.0 — Multi-agent fleet view.** Sub-agent spans are first-class in the explorer (Step 2). The natural next product is an org-level rollup showing every agent's runs across every customer.
