# Agorio v0.4 вҖ” Plan & Analysis

## Overview

v0.4 transforms Agorio from a single-merchant SDK into a multi-merchant commerce platform with real-world connectivity, webhook-based order tracking, and a browser playground for instant developer onboarding.

**Release Date:** February 27, 2026
**Previous Version:** v0.3.1 (191 tests, 13 tools, 4 LLM adapters)
**New Version:** v0.4.0 (233 tests, 17 tools, multi-merchant architecture)

---

## Problem Analysis

### What v0.3 lacked

1. **Single-merchant limitation** вҖ” Agents could only interact with one merchant per session. No price comparison, no multi-store shopping.
2. **No real merchant connectivity** вҖ” Only MockMerchant worked. No adapter for real commerce platforms (Shopify, WooCommerce, etc.).
3. **No async order tracking** вҖ” After checkout, agents had no way to receive shipping updates. Polling was the only option.
4. **No interactive demo** вҖ” The landing page showed code, but developers couldn't experience the agent without cloning the repo and running it locally.
5. **Limited shopping tools** вҖ” 13 tools covered basics but missed price comparison, multi-merchant switching, reviews, and discounts.

### Why these matter for adoption

- **Multi-merchant** is the core value proposition of an "agentic shopping" SDK. Without it, Agorio was just a checkout automation wrapper.
- **Real merchant support** (Shopify) proves the SDK works beyond mock servers, which is critical for developer trust.
- **Webhooks** enable event-driven architectures вҖ” the standard for production commerce.
- **Playground** removes the biggest friction point in developer adoption: "let me see it work before I invest time."

## Implementation Plan (5 Phases)

### Phase 1: Real Merchant Connectivity вң…

**Goal:** Connect to real Shopify stores via the Storefront API.

**Changes:**
- `src/adapters/shopify-adapter.ts` вҖ” ShopifyAdapter class wrapping Shopify Storefront GraphQL API
- `src/agent/shopping-agent.ts` вҖ” Adapter registry, auto-detection by domain pattern (`*.myshopify.com`)
- `src/types/index.ts` вҖ” `MerchantAdapter` interface, `ShopifyConfig` type

**Key decisions:**
- Used Storefront API (public, read-only catalog + checkout) rather than Admin API (requires OAuth, overkill for agents)
- Auto-detection by domain pattern keeps the developer API clean вҖ” just pass the domain
- Adapter pattern allows future WooCommerce, BigCommerce, etc.

### Phase 2: Multi-Merchant Architecture вң…

**Goal:** Support multiple merchants per agent session with isolated state.

**Changes:**
- `src/agent/shopping-agent.ts` вҖ” `merchants` Map keyed by domain, per-merchant cart/session state, new `switch_merchant` tool
- `src/llm/tools.ts` вҖ” 4 new tools: `switch_merchant`, `compare_prices`, `get_product_reviews`, `apply_discount_code`
- `src/types/index.ts` вҖ” `MerchantState` type, multi-merchant `AgentOptions`

**Key decisions:**
- Each merchant gets isolated state (cart, checkout session, capabilities) to prevent cross-contamination
- The LLM chooses when to switch merchants вҖ” no hardcoded orchestration
- Price comparison is a first-class tool rather than an LLM reasoning task (more reliable)

### Phase 3: Bug Fixes & New Tools вң…

**Goal:** Harden the agent loop and add missing commerce tools.

**Changes:**
- Fixed tool count assertions across test suite (12 вҶ’ 16 tools)
- Added `get_product_reviews` tool implementation
- Added `apply_discount_code` tool with percentage/fixed/free-shipping discount types
- Agent `executeTool()` switch covers all 16 tools

### Phase 4: Webhook Support вң…

**Goal:** Enable async order tracking via HMAC-signed webhook callbacks.

**Changes:**
- `src/webhook/webhook-server.ts` вҖ” New WebhookServer class (Express-based, HMAC-SHA256 verification)
- `src/mock/mock-merchant.ts` вҖ” Webhook subscription endpoints, `simulateOrderLifecycle()` (confirmed вҶ’ shipped вҶ’ delivered)
- `src/llm/tools.ts` вҖ” `subscribe_order_updates` tool (17th tool)
- `src/agent/shopping-agent.ts` вҖ” `webhookUrl`/`webhookSecret` options, tool implementation
- `tests/webhook.test.ts` вҖ” 15 new tests

**Key decisions:**
- HMAC-SHA256 signature verification follows industry standard (Stripe, Shopify, GitHub all use this)
- Lightweight Express server keeps dependencies minimal вҖ” developers bring their own framework in production
- MockMerchant simulates full order lifecycle with timed transitions for realistic testing
- Webhook subscription is an agent tool (not automatic) so the LLM decides when tracking is needed

### Phase 5: Browser Playground & Landing Page вң…

**Goal:** Interactive playground for zero-setup developer onboarding.

**Changes:**
- `site/app/playground/page.tsx` вҖ” Complete rewrite from pre-scripted replay to interactive agent
- `site/components/PlaygroundPreview.tsx` вҖ” Animated preview section on landing page
- `site/app/page.tsx` вҖ” Added PlaygroundPreview between Features and Tools sections
- `site/components/Hero.tsx` вҖ” v0.4 badge, multi-merchant code example, "Try Playground" CTA
- `site/components/StatsBar.tsx` вҖ” Updated to 233 tests, 17 tools
- `site/components/Tools.tsx` вҖ” All 17 tools listed
- `site/components/Features.tsx` вҖ” v0.4 feature cards (Multi-Merchant, Shopify, Webhooks, Observability)
- `site/components/CodeBlock.tsx` вҖ” Added ShopifyAdapter/WebhookServer to syntax highlighting

**Playground architecture:**
- Runs entirely client-side (no API keys, no backend) using Next.js static export
- In-browser product catalog mirrors `src/mock/fixtures.ts` (10 products)
- Client-side agent engine with intent detection (buy, browse, compare, review, discount, track)
- Tool execution against in-memory data with realistic timing/delays
- Free-form text input with suggestion buttons for discovery
- Real-time cart, order, and stats panels
