# Agorio v0.6 — Agorio Cloud MVP Plan

> **Status: Shipped (MVP) — v0.6.0, May 2026.** This document is now a historical artifact. See [`docs/ROADMAP.md`](ROADMAP.md) for the current plan and [`docs/cloud-setup.md`](cloud-setup.md) for the user-facing guide.
>
> **Original target:** Q3 2026 (~6–8 weeks of work)
> **Theme:** Replace the now-MIT plugins as the paid wedge with a hosted observability + control plane. Pro tier unlocks Agorio Cloud.

## Context

v0.5 relicensed the 5 governance plugins MIT and shipped them on npm, removing the "paid plugin" wedge. The Pro tier ($149/yr) needed a new product, and the SDK already emitted OTel-compatible spans, structured logs, and per-run usage stats — primitives that mapped naturally onto a hosted dashboard.

v0.6 wires those primitives to an ingestion endpoint and renders the data on a customer-facing dashboard at `cloud.agorio.dev`. The license key (existing billing anchor) stays in `site/`; per-environment API keys (new) gate the Cloud ingestion path.

## Goals

1. Ship `agorioCloud({ apiKey })` SDK helper that spreads into `AgentOptions`
2. Stand up a sibling Next.js app at `cloud.agorio.dev` for the dashboard
3. Persist runs/spans/logs in Neon with proper tenant isolation
4. Issue and revoke API keys from the existing `/dashboard`
5. Onboarding flow: Stripe checkout → license key → API key → first trace in <5 s
6. Reposition `/pricing` from "Cloud early access" to "available now"

## Out of scope (deferred to v0.6.1 and later)

- Hosted approval-workflow webhook receiver with click-to-approve UI (needs a new SDK primitive — the agent currently has no way to receive approval push-backs mid-run)
- Hosted mock merchants gated by license key (URL-accessible UCP/ACP/MCP merchants for CI pipelines)
- Fleet view / org-level rollup (requires an `orgs` table and a customer→org join)
- Stale-run sweeper (marks runs that never receive `run_end` as `failure`)
- Promotion of `db/` and `lib/auth-server.ts` into a `shared/` workspace package
- Switching the existing inlined `text({ enum })` columns on `customers` to true `pgEnum` types

Add 3 pgEnums and 4 tables:

```ts
pgEnum('api_key_env',     ['dev', 'prod', 'test'])
pgEnum('trace_status',    ['in_progress', 'success', 'failure'])
pgEnum('trace_log_level', ['debug', 'info', 'warn', 'error'])

traceLogs   { id, runId FK (cascade), level, message, data jsonb, timestamp }
```

Indices: `(customer_id, started_at desc)` on `traceRuns`; `(run_id)` on `traceSpans` and `traceLogs`; `(customer_id)` on `apiKeys`.

**Schema is also copied** to `cloud/db/schema.ts` with a "keep in sync" header (see Risks).

### Step 2 — SDK changes (~3 days)

**Files modified:**
- `src/types/index.ts` — add `onComplete?: (r: AgentResult) => void | Promise<void>` to `AgentOptions`.
- `src/agent/shopping-agent.ts` — call `await this.options.onComplete?.(result)` at 5 exit points: `run()` success (L204), `run()` max-iter (L256), `runStream()` success (L361), `runStream()` max-iter (L431), `runStream()` error path (L441). Extend `buildResult(success, answer, error?)` so the error path can produce a proper `AgentResult` with `result.error` populated.

**Files created:**
- `src/cloud/types.ts` — wire-format types: `SpanRecord`, `LogRecord`, `RunStartPayload`, `RunEndPayload`, `IngestBatch`, `IngestBatchType`.
- `src/cloud/index.ts` — `agorioCloud(opts)` returning `{ tracer, onLog, onStep, onComplete, beginRun, shutdown }` ready to spread into `AgentOptions`.

**Helper behavior:**
- Wrapper `AgentTracer` records `Date.now()` at `startSpan()` and emits a complete `SpanRecord` from `.end()` (the core `AgentSpan` interface carries no timestamps).
- When the outer `agent.run` / `agent.runStream` span opens, automatically POST a `run_start` batch — callers don't need to call `beginRun()` themselves.
- In-memory buffering, flushed on size threshold (default 25) or interval (default 1 s, `setInterval` is `.unref()`-ed). Final drain happens in `onComplete()`.
- `fetch(..., { keepalive: true })` with `Authorization: Bearer <apiKey>`. **All network errors are swallowed** with `console.warn` — they must never break a running agent.
- Trace ID generated client-side via `crypto.randomUUID()`. For v0.6 `runId === traceId`; field kept distinct on the wire to reserve room for multi-run traces later.
- `beginRun(task)` exported as an escape hatch for callers that don't use the `run()` lifecycle.

**Files modified for export:**
- `src/index.ts` — re-export `agorioCloud`, `AgorioCloudOptions`, `AgorioCloudHandle`, and wire types.
- `package.json` — add `"./cloud"` to `exports`, bump version to `0.6.0`.

**Verification:** `npm run typecheck` clean, new `tests/cloud.test.ts` (5 tests) passes — see Step 7.

### Step 4 — Ingestion endpoint (~1 day)

**File created:** `cloud/app/api/ingest/route.ts`

```
POST /api/ingest
Authorization: Bearer agorio_sk_<env>_<32hex>
Body: IngestBatch
Response: 202 Accepted (no body) on success, 401 on bad/revoked key, 400 on malformed batch
```

Logic:
1. Extract Bearer token → 401 if missing.
2. `lookupApiKey(token)` — 60 s in-memory TTL cache backed by `select * from api_keys where key = ? and revoked_at is null`. 401 if not found.
3. `touchLastUsed(apiKeyId)` — fire-and-forget update, 60 s debounce per key.
4. Switch on `batchType`:
   - `run_start` → insert `traceRuns` row with `onConflictDoNothing(traceRuns.id)`.
   - `spans` → bulk insert.
   - `logs` → bulk insert.
   - `run_end` → update `traceRuns` with totals, status, finalAnswer/error, endedAt.
5. Wrapped in try/catch; never throws.

`export const runtime = 'nodejs';`

## Risks (and how each was handled)

- **`AgentSpan` carries no start/end timestamps.** Wrapper tracer in `agorioCloud()` records start time in a closure and computes duration in `.end()`. No core-SDK span change required.
- **`AgentOptions` had no completion callback.** Added `onComplete` (3-line, backward-compatible change). `beginRun()` retained as an escape hatch.
- **Network failures must not break agents.** Helper catches all `fetch` errors and emits `console.warn` only. Asserted by test #4.
- **Two `node_modules/drizzle-orm` installs produce nominally-incompatible types when sharing a schema across apps.** Schema file is duplicated between `site/db/schema.ts` and `cloud/db/schema.ts` with a `KEEP IN SYNC` header. Migrations stay owned by `site/`. Consolidation into a `shared/` workspace package is tracked for v0.7.
- **API key leakage in logs.** Only `apiKeyId` (FK) is stored on `traceRuns`. UI masks keys to first 16 chars (`keyPrefix`) — the full key is shown exactly once in the create-form reveal card.
- **Neon HTTP per-request cost for `last_used_at` updates.** Fire-and-forget update, 60 s debounce per key, never blocks the 202 response.
- **Auth-handler duplication between `site/` and `cloud/`.** Documented with `// keep in sync` comments. Promotion to `shared/` deferred to v0.7.
- **Run never receives `run_end` (process crash).** Status stays `in_progress` indefinitely. A server-side sweeper that marks >1 h stale runs as `failure` is on the v0.6.1 list.

## What v0.6 unblocks

- **v0.6.1 — Hosted approval receiver.** The Cloud app now has authenticated tenants and Neon-backed storage. Adding a `/api/approve/[requestId]` endpoint + a new SDK primitive for the agent to poll/await approval is contained.
- **v0.6.1 — Hosted mock merchants.** Wrap `MockMerchant` / `MockAcpMerchant` / `MockMcpMerchant` behind a `cloud.agorio.dev/mock/<tenant-id>/...` route, gated by the same API-key lookup.
- **v0.7 — Procurement reference agent.** End-to-end demo can now point at a real production-grade observability surface.
- **v0.8 — EU AI Act audit exports.** `trace_runs` / `trace_logs` are the data layer; the export module just needs a PDF/CSV serializer.

## Post-launch patches (May 16, 2026)

After the initial 0.6.0 SDK publish + cloud deployment, a sequence of fixes shipped without an SDK version bump (cloud / site infrastructure only).

### Auth surface redesign

The initial `/login` route used `<AuthView path="sign-in" />` directly. That broke for two reasons:

1. **better-auth-ui hard-codes `basePath` to `/auth`** and generates internal links like `/auth/sign-up` and `/auth/forgot-password`. Without those routes existing, the sign-in page's own footer links 404'd.
2. **Cloud's layout was missing `NeonAuthUIProvider`** so `<AuthView>` silently rendered nothing (the component requires the provider as an ancestor).

Replaced both apps' `/login` page with `redirect('/auth/sign-in')` (backward-compat) and added a single dynamic route `app/auth/[pathname]/page.tsx` that renders `<AuthView pathname={pathname} />` for any auth view (sign-in, sign-up, forgot-password, reset-password, verify-email, callback). Added `cloud/components/Providers.tsx` mirroring site's; both pass `useRouter().push/replace` + Next `Link` into `NeonAuthUIProvider` for client-side nav, and force `defaultTheme="dark"`.

### Cross-subdomain session sharing

Added `cookies.domain: '.agorio.dev'` to both `auth-server.ts` files, gated on `VERCEL_ENV === 'production'` so localhost / preview deploys keep working. Sessions now persist across `agorio.dev` ↔ `cloud.agorio.dev`.

**Required Neon Auth config change**: `cloud.agorio.dev` had to be added to the project's **Trusted domains** list in the Neon Console (under Auth → Settings). Without it, the better-auth `originCheckMiddleware` on the Neon backend rejects requests with `{"code":"INVALID_ORIGIN"}` 403. This is *not* configurable from the app code — `createNeonAuth({ baseUrl, cookies })` has no `trustedOrigins` field. Documented in `docs/v0.6-release-checklist.md` under "DNS & Neon dashboard."

### Brand-native auth visuals

The default better-auth-ui forms render with shadcn/ui's neutral grayscale tokens which were invisible on Agorio's dark navy background. Mapped the library's `--neon-*` tokens (not the shadcn aliases — those collide with the project's brand variables of the same name) directly to our brand values in both `globals.css` files. Added `className="dark"` on `<html>` in both layouts to prevent FOUC. Wrapped `<AuthView>` in the same macOS-style terminal-frame aesthetic used elsewhere on the site (red/yellow/green dots + monospaced filename, e.g. `~/agorio/auth/sign-in.ts`), with per-element `classNames` overrides for the primary button (cyan→teal gradient), inputs (code-bg surface, cyan focus ring), labels (uppercase mono), title (JetBrains Mono).

### API-key management migrated to Cloud

Original plan put API-keys under `site/app/dashboard#api-keys`. That created a cross-domain UX hop for Cloud users (the keys are *used* by the Cloud SDK helper, so the management surface should live with the consumption surface). Built `cloud/app/api-keys/{page.tsx,actions.ts,CreateApiKeyForm.tsx}` mirroring site's UX but with `redirectTo` updated and using cloud's local Drizzle client. CloudNavbar now points `API keys` at the in-app route; a separate muted "Billing ↗" link goes out to `agorio.dev/dashboard` for license-key / Stripe portal management. `/traces` empty state rewritten as a numbered 3-step quickstart with a direct `/api-keys` CTA. Onboarding success page updated to land new subscribers on `cloud.agorio.dev/api-keys` instead of bouncing to the site dashboard.

Site's `/dashboard#api-keys` section is intentionally retained as a working secondary surface — anyone with existing onboarding emails / bookmarks still sees a functional UI; it's just no longer canonical.

### Middleware vs server actions

Cloud's middleware was initially configured with `matcher: ['/traces/:path*', '/api-keys', '/api-keys/:path*']`. The `/api-keys` entries broke server-action POSTs: Next.js 15 sends action bodies as `POST /api-keys` to the page route, and Neon Auth middleware's response augmentation made the client see "An unexpected response was received from the server." Removed `/api-keys` from the matcher; page-level `getCurrentCustomer()` and action-level `auth.getSession()` provide sufficient protection. Final matcher: `['/traces/:path*']` only.

### Env-var corruption bug (one-time)

When bulk-copying env vars from `agorio-site` to `agorio-cloud` via `vercel env pull` + re-piping to `vercel env add`, the pull's JSON-style escape format put a literal `\n` (two characters) at the end of each value. The cloud SDK proxy then constructed URLs like `…/auth\n/sign-in/email`, getting 404 from Neon. Re-set the three corrupted vars (`DATABASE_URL`, `NEON_AUTH_BASE_URL`, `NEON_AUTH_COOKIE_SECRET`) from clean source. Lesson now documented in `docs/v0.6-release-checklist.md` — don't round-trip values through `vercel env pull`.

### Manual customer comping

For testing without Stripe checkout, comped the owner's email by direct DB insert: `INSERT INTO customers (stripe_customer_id, email, license_key, status, plan) VALUES ('manual_comp_…', '<email>', 'agorio_pro_…', 'active', 'enterprise')`. Cleaner long-term path is auto-creating a `plan: 'free'` customer row on first auth (deferred to v0.6.1 — see roadmap).
